---
title: "Publish via HLS, MPEG-TS, and RTP"
description: "Ingest streams into MediaMTX from HLS servers, MPEG-TS over UDP or Unix socket, and RTP over UDP."
---

In addition to client-push protocols, MediaMTX can pull streams from HLS servers and ingest connectionless streams sent over UDP (MPEG-TS and RTP). These are configured as path sources in `mediamtx.yml`.

## HLS cameras and servers

HLS (HTTP Live Streaming) works by splitting a stream into segments and serving them with a playlist file (`.m3u8`) over HTTP. MediaMTX can connect to any HLS source — including Low-Latency HLS, MP4-based HLS, and legacy HLS — and expose the stream locally.

**Supported codecs**

| Type | Codecs |
|---|---|
| Video | AV1, VP9, H265, H264 |
| Audio | Opus, MPEG-4 Audio (AAC) |

Configure an HLS source by pointing to the playlist URL:

```yaml
paths:
  proxied:
    # URL of the playlist, in the format http://user:pass@host:port/path
    source: http://original-url/stream/index.m3u8
```

The ingested stream is available at `/proxied`.

## MPEG-TS

MediaMTX can ingest MPEG-TS streams delivered over UDP packets or Unix sockets.

**Supported codecs**

| Type | Codecs |
|---|---|
| Video | H265, H264, MPEG-4 Video (H263, Xvid), MPEG-1/2 Video |
| Audio | Opus, MPEG-4 Audio (AAC), MPEG-1/2 Audio (MP3), AC-3 |
| Other | KLV |

### MPEG-TS over UDP

Configure a path that listens for incoming UDP MPEG-TS packets:

```yaml
paths:
  mypath:
    source: udp+mpegts://238.0.0.1:1234
```

Where `238.0.0.1` is the IP address to listen on. A multicast IP causes MediaMTX to join the multicast group on the default interface chosen by the OS.

To specify a multicast interface explicitly:

```yaml
paths:
  mypath:
    source: udp+mpegts://238.0.0.1:1234?interface=eth0
```

To restrict which sender IP is accepted:

```yaml
paths:
  mypath:
    source: udp+mpegts://0.0.0.0:1234?source=192.168.3.5
```

Then push a stream to the configured address with FFmpeg:

```bash
ffmpeg -re -stream_loop -1 -i file.mp4 -c copy -f mpegts 'udp://238.0.0.1:1234?pkt_size=1316'
```

Or with GStreamer:

```bash
gst-launch-1.0 -v mpegtsmux name=mux alignment=1 ! udpsink host=238.0.0.1 port=1234 \
videotestsrc ! video/x-raw,width=1280,height=720,format=I420 ! x264enc speed-preset=ultrafast bitrate=3000 key-int-max=60 ! video/x-h264,profile=high ! mux. \
audiotestsrc ! audioconvert ! avenc_aac ! mux.
```

### MPEG-TS over Unix socket

Unix sockets are more efficient than UDP because they avoid the network stack entirely. Use the `unix+mpegts` scheme:

```yaml
paths:
  mypath:
    source: unix+mpegts:///tmp/socket.sock
```

Then push a stream with FFmpeg:

```bash
ffmpeg -re -f lavfi -i testsrc=size=1280x720:rate=30 \
-c:v libx264 -pix_fmt yuv420p -preset ultrafast -b:v 600k \
-f mpegts unix:/tmp/socket.sock
```

<Tip>
  Prefer Unix sockets over UDP when the publisher runs on the same host as MediaMTX — it reduces CPU overhead and eliminates packet loss.
</Tip>

## RTP

MediaMTX can ingest RTP streams delivered over UDP.

**Supported codecs**

| Type | Codecs |
|---|---|
| Video | AV1, VP9, VP8, H265, H264, MPEG-4 Video, MPEG-1/2 Video, M-JPEG |
| Audio | Opus, MPEG-4 Audio (AAC), MPEG-1/2 Audio (MP3), AC-3, G726, G722, G711 (PCMA, PCMU), LPCM |
| Other | KLV, MPEG-TS, any RTP-compatible codec |

Configure a path with both a `source` URL and an SDP description (`rtpSDP`). The SDP tells MediaMTX the codec, payload type, and clock rate of the incoming RTP stream:

```yaml
paths:
  mypath:
    source: udp+rtp://238.0.0.1:1234
    rtpSDP: |
      v=0
      o=- 123456789 123456789 IN IP4 192.168.1.100
      s=H264 Video Stream
      c=IN IP4 192.168.1.100
      t=0 0
      m=video 5004 RTP/AVP 96
      a=rtpmap:96 H264/90000
      a=fmtp:96 profile-level-id=42e01e;packetization-mode=1;sprop-parameter-sets=Z0LAHtkDxWhAAAADAEAAAAwDxYuS,aMuMsg==
```

Then push an RTP stream with FFmpeg:

```bash
ffmpeg -re -f lavfi -i testsrc=size=1280x720:rate=30 \
-c:v libx264 -pix_fmt yuv420p -preset ultrafast -b:v 600k \
-f rtp udp://238.0.0.1:1234?pkt_size=1316
```

<Note>
  `rtpSDP` must be a valid SDP document that accurately describes the codec and format of the incoming stream. An incorrect SDP will cause decoding failures or stream errors.
</Note>
