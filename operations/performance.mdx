MediaMTX ships with an integrated performance monitor that exposes [pprof](https://github.com/google/pprof)-compatible profiling data over HTTP. This lets you capture CPU profiles, analyze memory allocations, and list active goroutines without any external agents.

## Enabling the pprof endpoint

```yaml mediamtx.yml
# Allow clients to extract pprof-compatible performance reports.
pprof: true
# Address of the pprof listener.
pprofAddress: :9999
```

### TLS encryption

```bash
openssl genrsa -out server.key 2048
openssl req -new -x509 -sha256 -key server.key -out server.crt -days 3650
```

```yaml mediamtx.yml
pprofEncryption: true
pprofServerKey: server.key
pprofServerCert: server.crt
```

<Warning>
The pprof endpoint exposes detailed internals of the running process. Restrict access to trusted networks and avoid exposing it publicly. Access is governed by the `pprof` action in the authentication system; by default, only requests from `127.0.0.1` and `::1` are allowed.
</Warning>

## Capturing profiles

You need Go installed locally to use `go tool pprof`. The tool fetches the profile from the running server and displays a text report.

### CPU profile

Capture 15 seconds of CPU activity:

```bash
go tool pprof -text http://localhost:9999/debug/pprof/profile?seconds=15
```

Example output:

```
Fetching profile over HTTP from http://localhost:9999/debug/pprof/profile?seconds=15
Saved profile in /home/xxx/pprof/pprof.mediamtx.samples.cpu.003.pb.gz
File: mediamtx
Type: cpu
Duration: 15s, Total samples = 0
Showing nodes accounting for 70ms, 100% of 70ms total
      flat  flat%   sum%        cum   cum%
      30ms 42.86% 42.86%       30ms 42.86%  internal/runtime/syscall.Syscall6
      10ms 14.29% 57.14%       10ms 14.29%  runtime.(*consistentHeapStats).acquire
      10ms 14.29% 71.43%       10ms 14.29%  runtime.futex
      10ms 14.29% 85.71%       10ms 14.29%  runtime.mapIterStart
      10ms 14.29%   100%       10ms 14.29%  runtime.netpollblockcommit
```

The `flat%` column shows how much CPU time a function accounts for directly. The `cum%` column includes time spent in functions it calls.

### Heap (memory) profile

Analyze in-use memory allocations:

```bash
go tool pprof -text http://localhost:9999/debug/pprof/heap
```

### Goroutine dump

List all active goroutines and their call stacks:

```bash
go tool pprof -text http://localhost:9999/debug/pprof/goroutine
```

This shows how many goroutines are blocked in each function. A growing goroutine count over time may indicate a leak.

### Interactive browser UI

Open an interactive flame graph in your browser:

```bash
go tool pprof -http=:8080 http://localhost:9999/debug/pprof/profile?seconds=30
```

## Throughput and memory tuning

Beyond profiling, two configuration parameters have the largest effect on throughput and RAM usage.

### Write queue size

```yaml mediamtx.yml
# Size of the queue of outgoing packets.
# A higher value allows to increase throughput, a lower value allows to save RAM.
writeQueueSize: 512
```

`writeQueueSize` controls the number of packets that can be buffered in each outgoing queue before back-pressure is applied. Increase it when you observe dropped frames under high load; decrease it to reduce per-stream RAM consumption. Each unit holds one encoded packet, so the RAM cost scales with the number of concurrent readers.

<Tip>
Start by doubling `writeQueueSize` (e.g., `1024`) when streams drop frames under load, then profile heap usage to confirm memory remains acceptable.
</Tip>

### UDP read buffer size

```yaml mediamtx.yml
# Size of the read buffer of every UDP socket.
# This can be increased to decrease packet losses.
# It defaults to the default value of the operating system.
udpReadBufferSize: 0
```

When `udpReadBufferSize` is `0`, the OS default is used (typically 212 KB on Linux). If you observe packet loss on high-bitrate UDP sources (RTP, SRT, WebRTC), increase this value. A value of `4194304` (4 MB) is a common starting point for high-throughput deployments.

On Linux you can also raise the system-wide limit:

```bash
sysctl -w net.core.rmem_max=4194304
```

## When to use each profile

<AccordionGroup>
  <Accordion title="CPU profile — high CPU usage">
    Use when the process consumes unexpected CPU. A 15–30 second capture during peak load gives the most representative sample. Look for functions with high `flat%` — these are direct CPU consumers.
  </Accordion>
  <Accordion title="Heap profile — high memory usage">
    Use when RSS grows over time or you suspect a memory leak. The `inuse_space` type shows what is currently allocated; `alloc_space` shows cumulative allocations since start.
  </Accordion>
  <Accordion title="Goroutine dump — stuck or leaking goroutines">
    Use when the server becomes unresponsive or connection counts diverge from expectations. A large number of goroutines blocked in the same function often indicates a deadlock or unbounded channel.
  </Accordion>
</AccordionGroup>
