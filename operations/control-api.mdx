The Control API lets you inspect server state, update configuration at runtime, and forcibly terminate active sessions â€” all over HTTP without restarting the process.

## Enabling the API

Add `api: true` to your `mediamtx.yml`:

```yaml mediamtx.yml
# Allow clients to control the server through the Control API.
api: true
# Address of the Control API listener.
apiAddress: :9997
```

The API listens on port `9997` by default. By default it is accessible only from localhost.

## Security

### TLS encryption

Enable HTTPS by generating a self-signed certificate and updating the config:

```bash
openssl genrsa -out server.key 2048
openssl req -new -x509 -sha256 -key server.key -out server.crt -days 3650
```

```yaml mediamtx.yml
apiEncryption: true
apiServerKey: server.key
apiServerCert: server.crt
```

### CORS and trusted proxies

```yaml mediamtx.yml
# Allowed CORS origins. Supports wildcards.
apiAllowOrigins: ['https://dashboard.example.com']

# IPs or CIDRs of proxies placed before the API server.
# These proxies may set X-Forwarded-For and X-Forwarded-Proto headers.
apiTrustedProxies: ['10.0.0.0/8']
```

### Authentication

API access is governed by the same authentication system as the rest of the server. The `api` action controls who may call the API.

With the default internal auth, requests from `127.0.0.1` and `::1` are allowed without credentials. To require credentials from all callers, define a user with the `api` permission:

```yaml mediamtx.yml
authInternalUsers:
  - user: admin
    pass: secret
    ips: []
    permissions:
      - action: api
```

Then pass credentials in every request:

```bash
curl -u admin:secret http://localhost:9997/v3/paths/list
```

<Note>
When using HTTP or JWT authentication modes, the `api` action is excluded from external auth checks by default (`authHTTPExclude` / `authJWTExclude`). Add it back if you want external validation for API calls.
</Note>

## Endpoints

All endpoints are prefixed with `/v3`. The server base URL is `http://localhost:9997`.

### Server info

<CodeGroup>

```bash GET /v3/info
curl http://localhost:9997/v3/info
```

</CodeGroup>

Returns the running version and start time of the server.

---

### Configuration

<CodeGroup>

```bash GET /v3/config/global/get
curl http://localhost:9997/v3/config/global/get
```

```bash PATCH /v3/config/global/patch
curl -X PATCH http://localhost:9997/v3/config/global/patch \
  -H 'Content-Type: application/json' \
  -d '{"logLevel": "debug"}'
```

</CodeGroup>

`PATCH /v3/config/global/patch` applies a partial update â€” only the fields you include are changed. Changes take effect immediately without a restart.

Manage individual path configurations:

<CodeGroup>

```bash GET /v3/config/paths/list
curl http://localhost:9997/v3/config/paths/list
```

```bash GET /v3/config/paths/get/{name}
curl http://localhost:9997/v3/config/paths/get/mystream
```

```bash POST /v3/config/paths/add/{name}
curl -X POST http://localhost:9997/v3/config/paths/add/mystream \
  -H 'Content-Type: application/json' \
  -d '{"source": "rtsp://camera.local/stream"}'
```

```bash DELETE /v3/config/paths/delete/{name}
curl -X DELETE http://localhost:9997/v3/config/paths/delete/mystream
```

</CodeGroup>

### Active paths

<CodeGroup>

```bash GET /v3/paths/list
curl http://localhost:9997/v3/paths/list
```

```bash GET /v3/paths/get/{name}
curl http://localhost:9997/v3/paths/get/mystream
```

</CodeGroup>

`/v3/paths/list` returns all active paths including their state, source type, and connected readers. Results are paginated; use the `page` and `itemsPerPage` query parameters to navigate large sets.

### Recordings

<CodeGroup>

```bash GET /v3/recordings/list
curl http://localhost:9997/v3/recordings/list
```

```bash GET /v3/recordings/get/{name}
curl http://localhost:9997/v3/recordings/get/mystream
```

```bash DELETE /v3/recordings/deletesegment
curl -X DELETE \
  "http://localhost:9997/v3/recordings/deletesegment?path=mystream&start=2024-01-15T10:00:00Z"
```

</CodeGroup>

`DELETE /v3/recordings/deletesegment` requires `path` and `start` as query parameters. The `start` value is the starting timestamp of the segment as returned by `/v3/recordings/list`.
